How this document is organised

This document has three parts. Read in order.

  1. Context & correction. A short note clarifying the current state of the codebase, because the CEO's roadmap document understates what is already built.
  2. Feature specifications (BA Docs). One spec per feature across the three portals. Each spec is self-contained so the BA can pick up any single feature and write tickets without reading the rest. Every spec has the same shape: purpose → actors → user stories → functional requirements → what is built → what is pending → acceptance criteria → dependencies → risks.
  3. MVP definition & sprint plan. What the smallest shippable version of UTHBUS actually is, then concrete two-week sprints for solo execution — you, doing everything. Six sprints to MVP, then a sketched post-MVP roadmap because anything planned past month four will be wrong.

Correction to the CEO inventory

What is actually in main today

The CEO's inventory dated June 2026 calls out an "absent admin control tower." That is no longer true. Several things have shipped since that document was written. This section is the canonical starting point for the BA.

Admin portal exists. Routes live under app/admin/ with login, bus verification, and bookings oversight. contexts/admin-auth-context.tsx gates access. The portal is functional — what it is missing is commerce (commission, settlement, disputes) and operator KYC documents, not the portal itself.
The CEO is right on three counts. (1) The e-ticket at app/booking/ticket/[id]/page.tsx:344 hard-codes "Online (UPI)" regardless of whether anything was paid — confirmed by direct file inspection. (2) Operators self-onboard and become Active immediately, without document verification. (3) No payment gateway, no money ledger.

What exists in code (verified)

  • User portal: Homepage, search, booking flow (passenger details → review/pay → ticket), e-ticket with print/PDF/WhatsApp share, My Bookings, Profile, Google sign-in, six SEO route landing pages, stubs for Billing/Credits/Notifications/Refer-a-Friend.
  • Admin portal: Login, bus verification queue (Pending / Approved / Rejected / Suspended) with Approve / Reject / Hold / Reactivate / Delete actions, platform-wide bookings list with search + status filter + cancel action.
  • Operator portal: Registration + login, counter dashboard, walk-in counter booking, fleet management (add/edit/delete buses), bookings, trips, routes, seat layouts, reports, notifications, profile, settings.
  • Services: BusService, ActiveBookingsService, three auth contexts (user / operator / admin), three API routes for token handling.
  • Data model: IBus, IActiveBooking, BusVerificationStatus (pending_verification / approved / rejected / suspended), BookingStatus, plus user/operator/admin profile types.

What does not exist in code (the real gaps)

  • Payments. No gateway integration. The "Confirm & Pay" button writes the booking and lies on the ticket.
  • Operator KYC. No document upload, no admin-side review of operator credentials. License field is collected but never validated.
  • Commission & settlement. No ledger, no commission capture, no settlement run, no statements.
  • Refunds & cancellation policy. Cancelling flips status only; no money flow, no time-based rules.
  • Photos. Bus photos are a field, but there is no working upload pipeline.
  • Notifications. Real notifications (email, SMS, WhatsApp, push) are not wired. The user-side notifications page is a stub.
  • Ratings & reviews. Not present.
  • Programmatic SEO. Six hand-built route pages exist; no generator for more.

Everything in the sprint plan rolls forward from this list, not from the CEO's "nothing exists" framing.

Side one

User Portal — feature specifications

User Portal

Traveler-facing surface. The customer arrives via search, books a seat, and receives an e-ticket. Mobile-first. The booking funnel is the single most important conversion path in the entire product.

USR-HOME

Homepage & Search Widget

working P1
ModuleUser Portal · Discovery
Primary fileapp/page.tsx, components/homepage.tsx
Primary actorAnonymous visitor
Depends onUSR-SEARCH

Purpose

Capture origin / destination / date from a visitor in the fewest possible clicks and route them into search results. Communicate the brand promise (safety, choice, verified operators) above the fold.

Actors

  • Anonymous visitor — arrives from organic search, paid, or direct.
  • Returning user — already authenticated, sees personalised entry.

User stories

As a traveler I want to enter From, To, and Date on the homepage so that I land directly on a list of buses for my route.
As a traveler I want to see popular routes so that I can browse without typing if I don't have a destination locked in.
As a first-time visitor I want to immediately understand what UTHBUS is so that I trust it enough to start a booking.

Functional requirements

  • R1City pickers must be searchable dropdowns sourced from the canonical city list (not free text) to prevent typos that break search.
  • R2Date picker must default to tomorrow, prevent past dates, and allow up to 60 days ahead.
  • R3Swap button must atomically swap From and To values.
  • R4Submitting the form must navigate to /search?from=&to=&date= with all three params present and URL-encoded.
  • R5Featured route cards must be clickable and deep-link into search with the corresponding params pre-filled and date defaulted to tomorrow.
  • R6Page must render under 2.5s LCP on a mid-range mobile on 4G (Core Web Vitals threshold).

Currently built

  • Homepage with search widget, hero, featured route cards, "Why UthBus?" section — components/homepage.tsx.
  • View All routes modal listing seven destinations.
  • Route insight modals with buses/day, fares, peak times, boarding points (static content).
  • Welcome promo modal for new users.

Pending

  • P2Replace static "buses/day" copy in route modals with a live count once we have real inventory; otherwise it ages poorly.
  • P2Add JSON-LD WebSite + SearchAction schema for sitelinks search box in Google.
  • P3Personalised "Recently searched" row for returning authenticated users.

Acceptance criteria

  • Submitting valid From/To/Date lands the user on /search with results rendered within 3s on staging.
  • Past dates cannot be selected; date 61+ days out is rejected with a tooltip.
  • Lighthouse mobile score ≥ 85 on Performance, SEO, and Accessibility.

Edge cases & risks

  • Same From and To submission must be blocked with inline validation.
  • City names with Devanagari fallback must not break URL encoding.
  • Hydration mismatch on date picker when SSR-rendered — verify with timezone Asia/Kathmandu.
USR-SEARCH

Search Results & Filters

working P1
ModuleUser Portal · Discovery
Primary fileapp/search/page.tsx
Primary actorAnonymous or authenticated traveler
Depends onBusService.searchAllBuses()

Purpose

Given a From / To / Date triple, return a ranked, filterable list of bus options that match. Push the user toward a single bus to open in seat-selection.

User stories

As a traveler I want to filter buses by AC, type, price, departure time, and amenities so that I can find a bus that fits my preferences and budget.
As a traveler I want to see how many seats remain on each bus so that I don't waste time clicking into sold-out options.

Functional requirements

  • R1Only buses with verificationStatus = "approved" AND status = "Active" may appear in results.
  • R2Each result card must show: operator name, bus name, type, AC/non-AC, amenity icons, departure & arrival time, journey duration, fare, seats remaining, and verified badge once KYC is shipped (see ADM-KYC).
  • R3Default sort: departure time ascending. Secondary sort options: fare low-high, fare high-low, rating high-low (post-rating launch).
  • R4Filters must update results without full page reload (client-side state).
  • R5"No buses found" empty state must offer adjacent-date suggestions (±1, ±2 days).
  • R6Buses unavailable on the requested date (via unavailableDates) must be filtered out at query time, not at render time.

Currently built

  • Search page rendering results from BusService.searchAllBuses().
  • Filter sidebar: bus type, AC, price range, amenities, departure time window.
  • Result card with details and "View Seats" CTA into seat selection.

Pending

  • P0Verify that the query filters out non-approved buses. Today the search may surface unverified operators — check BusService.searchAllBuses() and add a Firestore composite index on (verificationStatus, status, fromCity, toCity).
  • P1"Seats remaining" should compute from capacity - bookedSeats(date), not from a static field. Today this number may be wrong on busy buses.
  • P2Adjacent-date "no results" suggestions (R5).
  • P2Surface verified badge once KYC ships.
  • P2Surface rating + review count once ratings ship.

Acceptance criteria

  • Search results return in < 800ms p95 for a city pair with up to 100 buses.
  • An unverified or suspended bus does not appear in any result set, verified by QA with a controlled test operator.
  • Filters compose: e.g. AC + price < 1500 + departing 18:00-22:00 returns exactly the matching set.

Edge cases & risks

  • Buses with blockedSeats for the requested date must show reduced availability.
  • Inventory thinness today means most filters return zero. Plan empty-state copy carefully.
USR-SEAT

Seat Selection

working P1
ModuleUser Portal · Booking Flow (step 1)
Primary fileBus detail / seat selection screen, contexts/booking-context.tsx
Primary actorAuthenticated traveler
Depends onActiveBookingsService for live seat state

Purpose

Show the bus layout. Let the user pick available seats. Hold them while the user moves through passenger details and payment. Prevent double-booking under concurrency.

User stories

As a traveler I want to see the live seat map so that I can pick exactly the seat I want.
As a traveler I want the seat I picked to be reserved while I finish booking so that someone else doesn't snatch it from me mid-flow.

Functional requirements

  • R1Seat map must reflect: available, blocked (operator-blocked), booked (already sold), selected (current user), and held (someone else mid-flow).
  • R2Anonymous users must be prompted to log in before seat selection persists.
  • R3Selected seats must be held for the user for up to 10 minutes; expired hold returns the seat to the pool.
  • R4A final re-check at booking confirmation must validate the seat is still held by the same user before writing the booking.
  • R5Max seats per booking: 6 (configurable per operator, see OPR-FLEET).

Currently built

  • Live availability check — taken seats are blocked.
  • Final re-check before confirmation prevents double booking under normal load.
  • Booking context stores bus + selected seats across the funnel.

Pending

  • P1Explicit seat-hold record with TTL. Today the re-check protects against duplicates but seats are not visibly "held" to other concurrent users, which causes UX confusion ("why does it say my seat is gone after I clicked it?").
  • P2Gender preference where applicable (Nepal market expectation for some routes).
  • P2Visual differentiation between upper and lower deck for Sleeper buses (depends on expanded bus type taxonomy — see OPR-FLEET).

Acceptance criteria

  • Two users selecting the same seat within 100ms: one wins, the other receives a clear "seat just taken" message and is returned to the map.
  • A held seat that times out returns to "available" within 30s of TTL expiry.

Edge cases & risks

  • Multi-tab booking by the same user — second tab should not be allowed to confirm.
  • Hold TTL must survive user navigating away and returning.

USR-PAY

Review & Pay

broken P0
ModuleUser Portal · Booking Flow (step 3)
Primary fileapp/booking/review-pay/page.tsx
Primary actorAuthenticated traveler
Depends onPayment gateway integration (new), Promotions service (ADM-PROMO)

Purpose

Final confirmation step. Summarise the booking, apply promos, take payment, write the booking only on payment success. This is the most broken screen in the product today: it skips payment entirely.

Critical defect: The "Confirm & Pay" button writes a confirmed booking without any payment being captured. The next screen then claims the booking was paid by UPI. This must be fixed before any real users are routed in.

User stories

As a traveler I want to see the full fare breakdown, apply a promo, and pay with eSewa, Khalti, or card so that I have a real, trustworthy transaction.
As a traveler whose payment fails I want a clear retry path and my seats still held so that I don't lose my place.

Functional requirements

  • R1Fare breakdown line items: base fare × seats, service fee (if any), discount (promo), total.
  • R2Promo code field validates against the promotions service (see ADM-PROMO). Today only WELCOME50 is hard-coded — remove the hard-code and check against backend.
  • R3Payment method selector: eSewa, Khalti, Card (Stripe or equivalent), Pay at Counter (operator-allowed only).
  • R4"Confirm & Pay" initiates a payment intent and redirects to the gateway. Booking is written with status = "Pending" at intent creation, flipped to Confirmed only on webhook success.
  • R5Failed payments: booking remains Pending for a TTL (e.g. 15 min) with retry. After TTL it auto-cancels and releases seats.
  • R6Webhook handler must be idempotent — the same gateway event arriving twice must not double-book or double-charge.
  • R7"Pay at Counter" path (where allowed by the operator) writes the booking with paymentStatus = "unpaid_pending_counter" and renders a clearly different ticket (see USR-TICKET).

Currently built

  • Review screen with fare summary.
  • Single hard-coded promo WELCOME50 (flat NPR 50 off).
  • "Confirm & Pay" writes the booking as Confirmed immediately. No money flow.

Pending

  • P0Stop writing Confirmed without payment. Introduce paymentStatus on the booking model: unpaid_pending_counter | pending_gateway | paid | failed | refunded.
  • P0Integrate eSewa.
  • P0Integrate Khalti.
  • P1Integrate card payments for tourists (Stripe or local card gateway).
  • P1Payment intent model + webhook handlers (idempotent).
  • P1Replace hard-coded promo with promotions service driven by ADM-PROMO.
  • P2"Pay at Counter" path with operator opt-in.

Acceptance criteria

  • A booking only becomes Confirmed after a successful payment webhook OR after a counter operator marks it paid.
  • A failed payment leaves the seat held for the user to retry within the TTL.
  • Replaying the same payment webhook is a no-op (idempotency tested by QA).
  • No code path produces a "paid" ticket without a corresponding payment record.

Dependencies

  • Merchant accounts: eSewa, Khalti, card gateway. Procurement is a CTO-side blocker, not engineering — start now.
  • Settlement ledger (ADM-COMM) for downstream reconciliation.

Edge cases & risks

  • Webhook delivery delays — the user may have returned to the app before the webhook arrives. UI must poll or use a websocket fallback.
  • Refunds initiated before settlement vs. after — different ledger entries.

Operator Portal — feature specifications

Operator (Counter) Portal

Used by bus operators — the supply side of the marketplace. Manages fleet, takes walk-in bookings at the counter, tracks daily operations, and (once shipped) reconciles money. Strongest part of the codebase today, but missing the commerce layer that makes operators stay.

partial P0
ModuleOperator Portal · Account

Purpose

Onboard a new bus operator. Today this is broken: anyone with an email becomes an Active operator immediately. We need a verification gate before any of their buses can appear in user search.

Functional requirements

  • R5Approved operators receive a verified badge surfaced on USR-SEARCH and USR-TICKET.
  • R6Rejected operators see the rejection reason on login and can re-submit corrected documents.

Currently built

  • Registration + login pages with company name, license (free-text, optional), address.
  • No document upload. No KYC state. Operator becomes Active instantly.

Pending

  • P0Add kycStatus field to operator model with states above.
  • P0Document upload (Firebase Storage with rules) for company cert, license, PAN.
  • P0Gate user-facing listing on kycStatus = "approved" (already gated on bus-level verificationStatus — combine both checks).

First, the only thing that matters

What is the UTHBUS MVP?

"MVP" is one of the most abused words in product. For UTHBUS, here is the only definition that matters: the smallest version of the platform that can take real money from a real traveler, give them a truthful ticket, and pay the real operator their share. Everything else is post-MVP.

MVP in one sentence: A verified Nepali operator lists one bus → a real traveler finds it on uthbus.com → pays via eSewa → receives a ticket that is true → rides the bus → you, as platform, can show what you owe that operator and settle them. Anything not needed to make that loop work end-to-end is post-MVP.

What is IN the MVP

  • Honest e-ticket. Payment Mode rendered from real data, never hard-coded.
  • Operator KYC. Documents uploaded, admin approves, verified badge shows. "Verified operators" becomes true in software.
  • One payment method: eSewa. Most common in Nepal, simplest sandbox, fastest to ship. Khalti, cards, wallet — later.
  • Cancellation policy + manual refund. Policy enforced in UI; refund triggered by you from admin (manual eSewa refund through their dashboard). Automated refund is post-MVP.
  • Commission ledger (read-only). Every paid booking writes a ledger entry with gross / commission / net. Operator can see what they are owed. Actual payout happens outside the system (you bank-transfer them weekly). Automated settlement run is post-MVP.
  • Admin can step into a booking. Already built. Add reason fields, that is enough.

What is NOT in the MVP (and why)

  • Khalti, cards (Stripe), wallet. Cost vs benefit is wrong for a v0. Add the second payment method only after the first one shows traction.
  • Automated refunds. Solo, ten refunds a week is faster done manually than building a robust refund API. Build the manual flow well, automate later.
  • Automated settlement run. Same logic. Excel sheet + bank transfer + mark-paid on admin is enough until you have 10+ active operators.
  • Bus photo upload. Use brand-tier placeholder images by bus type for MVP. Real photos are post-MVP polish.
  • WhatsApp / email notifications. Manual WhatsApp from your business number is faster than building Wati integration solo. In-app + email-on-booking-confirmation is enough.
  • Ratings and reviews. Not needed to take the first 100 bookings.
  • Disputes module. Use a Google Form + WhatsApp until you actually have disputes worth ticketing.
  • Programmatic SEO, PWA, Sleeper layouts, promo engine. All post-MVP.
  • 2FA on admin, audit log UI. Audit log writes yes (cheap), UI no.
The rule: if a feature is not needed for "a traveler pays real money and rides a real bus, and you can pay the operator", it is not in the MVP. When in doubt, cut it. You can ship a feature in week 14; you cannot un-ship time spent building the wrong thing.

How you know MVP is done

  • You have at least one verified operator with one bus listed.
  • A real user (not you, not a friend) can complete a booking end-to-end on production.
  • The ticket they receive accurately reflects payment status and method.
  • You can show that operator, by Monday, exactly what they earned, what your commission was, and what you owe them — even if you settle them by manual bank transfer.
  • You have run one full booking-to-settlement cycle with real money. Even at NPR 500, it counts.

Solo reality check

Capacity, scope, and the math

A solo founder-developer is not 1/5 of a five-person team. You are also doing sales, operator acquisition, support, ops, and decisions. The honest math:

Your capacity

  • Realistic coding time: 25-30 productive hours per week. The other 25+ go to operator calls, content, support, decisions, taxes, life.
  • Per two-week sprint: ~50-60 hours of focused engineering work. Plan to this number, not to optimistic ones.
  • Solo penalty: every task takes ~1.3-1.5x longer than the same task with a team, because there is no one to bounce ideas, review your PR, or unblock you when you are stuck on a gateway sandbox.
  • Context-switch cost: doing backend + frontend + ops in the same sprint is slower than batching. Try to batch by surface: one week mostly backend, one week mostly frontend, in each sprint.

Sprint operating model (just you)

Build (Mon-Thu mornings)
~16 hours / week
  • Deep work on the current sprint's one big thing.
  • No meetings before 1pm if you can help it.
  • If you finish the sprint goal early, pull from "Pending" of the same feature spec — do NOT start a new feature.
Test & ship (Thu afternoon, Fri)
~8 hours / week
  • Self-QA the week's work against the requirement IDs (USR-PAY-R1, R2…).
  • Deploy to staging Friday. Production only when the full sprint is signed off by you.
  • Keep a personal regression checklist; run it before every prod deploy.
Ops & support (daily)
~6 hours / week
  • Triage Sentry errors and user messages daily, 30 min cap.
  • Do not let support eat the build block. Batch responses at lunch and end of day.
  • Anything not on fire goes into the backlog, not into today.
Plan & review (Friday afternoon)
~2 hours / week
  • Mini-retro: what shipped, what slipped, what to defer.
  • Confirm next week's single goal. Write it on top of the doc.
  • Don't replan past the next sprint — solo plans drift fast.

Rules that protect a solo developer from themselves

  1. One sprint, one goal. If you can't summarise the sprint in one sentence, it has too much in it.
  2. If a thing can be done manually, do it manually first. Refunds, settlements, KYC review, notifications — manual is fine for MVP. Automate only after volume justifies it.
  3. Never start two sprints' worth of work at once. "I'll just also…" is the solo killer.
  4. Cut before you stretch. If a sprint is slipping, cut from this sprint, do not push the next sprint's date.
  5. Don't refactor on the critical path to MVP. Ugly working code > pretty broken code. Refactor in Sprint 6+.
  6. Ship to staging every Friday, even if the goal isn't fully done. Avoids week-long branches that rot.
  7. Write down one decision per week. "Why I chose eSewa first" — one paragraph. Future you and the first hire will thank present you.
The single most important rule: ship MVP in 14 weeks. If by week 10 you are still on Sprint 3, the answer is to cut Sprint 5's scope, not to push to week 18. The cost of slow MVP > the cost of a smaller MVP.

Sprint 3 eSewa Payment (one gateway only) Goal: a real traveler can pay real money · 2-3 weeks (~75 hrs)
The one goal
Booking flow ends with a real eSewa transaction. Booking only flips to Confirmed on successful callback. The "Paid via eSewa" variant of the e-ticket renders with the real txn id.
Pre-sprint (do before week 1)
  • Get eSewa merchant credentials. This can take days to a week with their team. Start now, not on day 1 of the sprint. Sandbox creds first, prod later.
  • Read eSewa v2 ePay documentation cover to cover. Especially: signature scheme, callback URL format, transaction status verification endpoint.
  • Write down on paper: who initiates, who redirects, what callback carries, how to verify.
Week 1 · intent + initiate
  • Add Firestore collection paymentIntents: {id, bookingId, gateway: "esewa", amount, status: "initiated|success|failed", gatewayTxnId, createdAt, updatedAt}.
  • Update booking write path: on "Confirm & Pay", create paymentIntent + write booking with paymentStatus = "pending_gateway". Do NOT mark Confirmed yet.
  • Add seat-hold record: hold the seats for 15 min tied to the paymentIntent id. Expired holds release the seats.
  • Build the eSewa initiate handler: take amount, build signed payload, redirect to eSewa.
  • Test in eSewa sandbox: can you reach the eSewa page from your local app?
Week 2 · callback + idempotency
  • Build the eSewa success callback handler (in app/api/ or Cloud Function).
  • Signature verification first. Reject any callback that doesn't verify, log to Sentry. This is your security-critical path.
  • Verify the transaction with eSewa's status API as a second check (don't trust the callback alone — users can replay/forge).
  • Update paymentIntent → success, update booking → paymentStatus="paid", paymentMode="eSewa", paymentTxnId=<refid>, status="Confirmed". All in one Firestore transaction.
  • Idempotency: if the same gateway txn id arrives twice (it will, eSewa sometimes retries), the second call is a no-op. Test by replaying manually.
  • Build the failure callback handler. Mark intent failed, release seat hold, leave booking flippable to "Pay at Counter" fallback.
  • Front-end: update app/booking/review-pay/page.tsx — remove the auto-confirm path entirely. Add eSewa as the only paid option. Add "Pay at Counter" as the second option (writes Pending-Counter immediately).
  • "Payment pending" screen for the case where the user is redirected back but the callback hasn't arrived yet. Poll the paymentIntent status, max 60s, then show retry.
Week 3 (buffer + production)
  • Migrate to eSewa production credentials.
  • Take one real booking through (yourself, NPR 100, refund manually after).
  • Add Sentry alert: any callback that fails signature verification pages you.
  • Add a Cloud Function scheduled job: expire any paymentIntent stuck in "initiated" for >15 min (releases held seats).
Cut from this sprint
No Khalti. No card / Stripe. No refund automation (cancellation policy is just text on the ticket page for now). No "Pay at Counter" operator opt-in flag (allow it for everyone in MVP). No promo engine.
Sprint exit: v0.3-payments. One real eSewa transaction has cleared on production. Idempotency verified by replay. This is the hardest sprint — budget the buffer.
Sprint 4 Cancellation Policy + Manual Refund Goal: users can cancel, money flows back (you press a button) · 2 weeks (~50 hrs)
The one goal
A user with a paid booking can cancel through the app, see what refund they will get, and you have a single admin queue showing every pending refund. You process the refund in eSewa's portal yourself, then click "Mark refunded" — the system handles the rest.
Why manual
Automating eSewa refund through their API takes another sprint and they are flaky in the sandbox. At MVP volumes (1-20 refunds a week), you logging into eSewa and pressing refund is faster, cheaper, and more reliable. Automation goes in post-MVP.
Week 1 · policy + cancel UX
  • Cancellation policy as a config: tiered refund %s by hours-to-departure. Defaults: >24h = 90%, 6-24h = 50%, <6h = 0%. Per-operator override possible but not required for MVP.
  • Helper calculateRefund(booking) returning {refundable: boolean, amount: number, tier: string}.
  • USR-MY cancel button updates: show refund preview ("You will be refunded NPR 450 of NPR 500. Refund processes in 3-5 business days."), confirm dialog, then call cancel API.
  • Add booking status Cancelled_PendingRefund (or use existing Cancelled with a separate refundStatus field: none | pending | processed).
  • When user cancels: write refundStatus = "pending", store the calculated refund amount, free the seat immediately.
Week 2 · admin refund queue
  • New admin page /admin/refunds: list bookings with refundStatus = "pending". Show: PNR, user, operator, original amount, refund amount, paymentTxnId, booked date, cancel date.
  • "Mark refunded" action: opens a dialog asking for refund reference (the txn id from eSewa's portal after you process it), then writes refundStatus = "processed" + refundTxnId + refundProcessedAt + admin uid.
  • Email user on refund processed (Firebase "Trigger Email" extension again).
  • Add a small dashboard card on the admin index: "X refunds pending".
  • USR-MY refund-status display: show "Refund pending" / "Refunded on <date> (ref: <txnId>)".
Cut from this sprint
No automated gateway refund API. No partial-refund splits. No refund webhook (you're triggering manually). No reason-required field on cancel (optional).
Sprint exit: v0.4-refunds. A cancellation moves money back, even if you are the bottleneck. Time-to-refund target: same day if cancelled before 5pm.
Sprint 5 Commission Ledger → SHIP MVP Goal: you know what each operator earned. Then ship. · 2-3 weeks (~70 hrs)
The one goal
Every paid booking writes a ledger entry that breaks down gross / commission / net. The operator can see what they have earned and what you owe them on their portal. You can see the same on admin, and you settle them by bank transfer outside the system.
Why no automation
An automated settlement run with bank-API integration is a sprint of its own. Solo, with 1-5 operators, manual is fine. The ledger needs to be accurate; the payout itself is a few clicks in your banking portal each Friday.
Week 1 · ledger
  • Add Firestore collection ledgerEntries: {id, bookingId, operatorId, type: "earning|refund_reversal|adjustment", grossAmount, commissionAmount, netAmount, settledStatus: "unsettled|settled", settledAt, settlementId, createdAt}.
  • Config doc config/commission: {defaultRate: 0.10}. Per-operator override goes on operator doc as commissionRateOverride.
  • Helper computeCommission(grossAmount, operator) returning {grossAmount, commissionAmount, netAmount}.
  • On payment success (Sprint 3 callback): write a ledger entry with type=earning. Same Firestore transaction as the booking flip.
  • On refund processed (Sprint 4 admin action): write a reversal entry (negative gross, negative commission, negative net) referencing the original.
  • Test by hand on staging: 10 fake bookings + 2 refunds. Sum the ledger per operator. Numbers must reconcile.
Week 2 · operator + admin views
  • OPR-EARN page (/operator/counter/earnings or similar): three cards (this week / this month / all-time) showing gross, commission, net. Below: ledger table with PNR, date, gross, commission, net, settled status.
  • ADM-COMM admin page (/admin/settlements): one row per operator with unsettled balance. Click into row to see all unsettled ledger entries. "Mark settled" action: enter payout reference (your bank transaction id), date, and it marks all selected entries settled.
  • Add bank/eSewa/Khalti payout fields to operator profile (just text fields, you'll copy-paste when you do the transfer).
Week 3 · MVP regression + ship
  • Run your full personal regression checklist across user / operator / admin portals. Cross-check against requirement IDs of every P0/P1 feature.
  • Onboard your first operator end-to-end on production: register, upload KYC, you approve, they list one bus, you list it.
  • Take one real booking yourself, ride the bus if you can, settle the operator manually that week.
  • Write a short post-mortem doc: what worked, what was harder than expected, what to fix first post-MVP. This is the artifact for your future first hire.
  • Production deploy. Announce. Take real bookings.
Cut from this sprint
No automated settlement run. No reconciliation report. No CSV export (operator can take screenshots, you can manually pull from Firestore for the first few). No multi-currency. No per-route commission. No promo cost allocation. Anything that is not "ledger writes and two views" is post-MVP.
Sprint exit: v1.0-mvp. 🚀 MVP IS LIVE. First real operator is verified, first real booking is taken on prod, first real settlement is paid by you. From here, the plan changes — you'll know what hurts and what to do next.

After MVP ships

Post-MVP roadmap (sketched)

Anything planned in detail past month four is fiction. By the time MVP is live you will know which gaps hurt most — and the order below will be wrong. Use this as a menu, not a contract. Each sprint is still ~2-3 weeks solo. Pull one at a time, the one that is hurting MVP users the most.

The bigger decision at this point: hire someone. Solo can ship MVP. Solo cannot do MVP + sales + ops + post-MVP features at the same time without something breaking. The first hire is probably a frontend engineer (most volume of UI work below) or, if money permits, a full-stack mid-level who can pair with you. Re-evaluate at end of Sprint 5.

What could derail this

Risks & assumptions

Assumptions baked into the plan

  • eSewa merchant credentials are procured by end of Sprint 2. Start the application now — their team takes days to a week to respond. Sandbox first; production once Sprint 3 is ready.
  • You can dedicate ~25-30 productive engineering hours per week. If real life (illness, family, a Diwali week) eats into that, slide sprint boundaries — do not stretch the goal.
  • By end of Sprint 2 you have a real candidate first operator already in conversation, so KYC has a real subject to test on. Operator acquisition is its own workstream and not in this doc.
  • You will not pull yourself toward the mobile app, loyalty features, or programmatic SEO until MVP is shipped. Write the next idea down, put it in this doc's post-MVP menu, then close the tab.
  • The codebase is stable enough on main that small migrations (paymentStatus, kycStatus) won't break unrelated flows. Smoke-test before each migration.

Risks

1You are the single point of failure. Sick week = sprint slip. Mitigation: keep sprints to one goal so a one-week slip costs one feature, not the whole plan. Tag every prod deploy with a version so you can roll back without thinking.
2eSewa sandbox is flaky. Their docs are out of date in places, and callback IPs sometimes drift. Mitigation: budget a full week of buffer in Sprint 3 specifically for "things eSewa-shaped don't work". Have a backup plan to ship "Pay at Counter only" if eSewa slips by > 3 weeks.
3Scope creep on yourself. Solo devs are notorious for "while I'm in here, I'll also fix…". Mitigation: write the temptation down in the post-MVP menu, do not action it. Re-read the MVP definition every Monday.
4No code review. No second pair of eyes on payment / ledger code, which is where bugs are most expensive. Mitigation: write tests for the money paths (compute commission, refund reversal, idempotency). Also: re-read your own PR the next morning before merging — the gap of a night helps a lot.
5Data migrations on production with no team backup. If a backfill goes wrong on prod data, recovery is on you alone. Mitigation: Sprint 0 sets up daily Firestore exports for a reason. Every migration runs on staging twice before prod, and never on a Friday afternoon.
6Webhook reliability + idempotency. Money flows depend on webhook delivery and not double-processing. Build retry and idempotency from Sprint 3 day one, not as a "we'll harden later" item.
7Operator supply. The plan assumes a real operator is ready to be your first verified KYC by end of Sprint 2. If not, the verification flow tests on synthetic data and the first MVP transaction slips. Run operator acquisition in parallel.
8Burnout. Solo at 25-30 productive hours for 14 weeks straight is doable; at 50 hours, you crash by week 8. Protect the 30-hour ceiling. Take a real day off each week. The plan respects that already.
9Security rules per new collection. Every new Firestore collection (paymentIntents, ledgerEntries, kycDocs) needs explicit rules. Solo means no one reminds you. Put it on the per-sprint definition of done.
10You take the first booking and break it under load. Unlikely at MVP traffic, but the day a route goes viral, payment + seat-hold + ledger writes all spike. Mitigation: build the booking write inside a Firestore transaction (already implied above). Add a Sentry "transaction failed" alert.

Reference

Glossary & conventions

  • PNR. Passenger Name Record — the unique booking reference shown on the ticket.
  • KYC. Know Your Customer — in our context, the operator-side document verification.
  • Ledger entry. One immutable financial record per money event (booking paid, refund, commission, settlement).
  • Settlement run. The job that aggregates unsettled ledger entries per operator into a payable. For MVP this is manual: you query the ledger, total it, bank-transfer the operator, then click "mark settled" in admin.
  • Verified badge. UI mark shown next to an operator on user surfaces once their KYC is approved.
  • P0 / P1 / P2 / P3. Priority: P0 = blocks revenue or trust (must ship before MVP); P1 = needed for a healthy MVP; P2 = needed to scale; P3 = nice-to-have.
  • Feature IDs. Stable identifiers, used in tickets and test cases. Format: {PORTAL}-{SHORT}. Requirement IDs: {FEATURE-ID}-R{n}.
  • Sprint length. Two weeks for MVP sprints (except Sprint 0 = 1 week). Some sprints budget a third buffer week for solo realities. Demo + mini-retro on the last Friday.
  • Definition of Done (solo edition). Code merged to main, deployed to staging, you walked through your personal regression checklist mapping to feature requirement IDs, Sentry has no new errors after 24h, and the related feature spec in this doc has been updated (move items from Pending to Built).
  • MVP. The smallest version that can take real money, give a true ticket, and pay an operator. Not a smaller version of v1.0 — a different product that does only the loop. See the MVP section.
  • Manual-first. The solo principle: build the manual flow (refunds, settlements, notifications) before the automation. Cheaper, faster, more reliable at MVP volumes. Automate when manual hurts.
One-line summary. Ship eSewa-only, manually-settled, KYC-verified MVP in 14 weeks. Then take real bookings, see what hurts, and use the post-MVP menu as a guide — not a contract. By Sprint 6 or 7, hire someone.
